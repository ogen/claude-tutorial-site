    第二十五章：安全性與合規¶

    在導入前，先回答這個問題¶

    在組織裡推廣 Claude 之前，最常被法務或資安團隊問到的問題是：「我們輸入的資料，Anthropic 會不會拿去訓練模型？」

    直接的答案是：Enterprise 方案預設不用於模型訓練。但這個議題涉及的細節比較多，這一章會完整說明 Claude 如何處理你的資料，以及有哪些合規認證可以作為依據。

    資料處理政策¶

    資料是否用於訓練？¶

    方案 預設行為 可否調整
    Free / Pro 可能用於訓練改善模型 可在設定中關閉
    Team 不用於訓練 —
    Enterprise 不用於訓練 —
    API 呼叫 不用於訓練 —

    Team 以上方案和 API 呼叫，Anthropic 預設不會把你的對話資料用於模型訓練。

    資料保留期限¶

    • 標準：對話資料保留 30 天（用於安全性監控），之後自動刪除
    • Enterprise 零保留模式（Zero Retention）：對話結束後立即不保留在 Anthropic 伺服器，適合高度敏感的金融、醫療場景

    啟用零保留模式後，部分功能（如 Memory 記憶系統）會無法使用，因為這些功能本身需要儲存資料。

    資料傳輸安全¶

    • 所有傳輸使用 TLS 1.2 以上加密
    • 靜態資料（at rest）使用 AES-256 加密
    • Anthropic 的基礎架構部署於 AWS us-east-1（美國東部）

    合規認證¶

    Anthropic 已取得或正在維護的認證，可作為內部合規評估的依據：

    認證 / 標準 說明 適用場景
    SOC 2 Type II 資訊安全控制獨立稽核 大多數企業的基本要求
    ISO 27001 國際資訊安全管理系統標準 跨國企業、歐洲市場
    GDPR 合規 歐盟個人資料保護法規 涉及歐盟用戶的組織
    HIPAA BAA 美國醫療資訊保護（需簽訂 BAA） 醫療健康產業
    CCPA 加州消費者隱私法 涉及加州用戶

    最新認證狀態請以 Anthropic 官方信任中心（https://trust.anthropic.com）為準，認證內容可能隨時更新。

    SSO 安全設定¶

    強制 SSO 登入¶

    Enterprise 方案可以強制所有用戶只能透過 SSO 登入，禁止直接用 email + 密碼登入，降低帳號被盜的風險：

    管理控制台 → Security → Authentication
→ 開啟「Require SSO」
→ 設定允許的 IdP（如 Okta、Azure AD）

    多因素驗證（MFA）¶

    如果你的 IdP 已有 MFA 政策，啟用 SSO 後 Claude 會自動繼承這個保護機制，不需要在 Claude 端另外設定。

    Session 管理¶

    管理控制台 → Security → Session Settings

    可設定的項目： - Session 有效期（建議 8-24 小時，依安全政策決定） - 閒置超時（建議 1-2 小時） - 強制重新驗證的條件

    稽核日誌（Audit Logs）¶

    Enterprise 方案提供完整的操作日誌，讓資安團隊能追蹤所有帳號活動。

    記錄的事件類型¶

    • 用戶登入 / 登出
    • Projects 的建立、修改、刪除
    • 用戶權限異動
    • 管理員操作記錄
    • API Key 的建立與撤銷

    匯出與整合¶

    稽核日誌支援匯出為 JSON 格式，可以整合進你的 SIEM 系統（如 Splunk、Microsoft Sentinel）：

    {
  "timestamp": "2026-06-22T08:30:00Z",
  "event_type": "user.login",
  "user_id": "usr_abc123",
  "user_email": "alice@company.com",
  "ip_address": "203.0.113.42",
  "auth_method": "sso",
  "result": "success"
}

    保留期限¶

    稽核日誌預設保留 90 天，Enterprise 方案可依需求延長至 1 年或更長。

    敏感資料的使用建議¶

    即使 Claude 的資料安全符合企業要求，在使用上仍建議養成幾個好習慣：

    應避免直接貼入的資料¶

    • 客戶的個人識別資訊（姓名＋身分證號、信用卡號等組合）
    • 未公開的財務預測或 M&A 資訊
    • 密碼、API Key、私鑰

    處理敏感資料的替代做法¶

    # 不好：直接貼入真實資料
prompt = f"分析以下客戶資料：{customer_pii_data}"

# 較好：先脫敏再送給 Claude
anonymized = anonymize(customer_pii_data)
prompt = f"分析以下匿名資料：{anonymized}"

    建立內部使用規範¶

    建議在 Intranet 或 Confluence 建立一份「Claude 使用規範」，明確列出： - 哪些資料可以直接使用 - 哪些資料需要脫敏後使用 - 哪些任務完全禁止使用 Claude 處理

    實作練習¶

    1. 合規需求盤點：列出你的組織目前需要符合的法規或認證（如 ISO 27001、GDPR、HIPAA），對照本章的認證表確認 Claude 是否已滿足，若有疑問記下來向 Anthropic 業務窗口確認。

    2. 稽核日誌設定：若有 Enterprise 方案，進入管理控制台的稽核日誌頁面，下載最近 7 天的日誌，確認格式是否符合你的 SIEM 整合需求。

    3. 敏感資料分類練習：把你工作中常輸入給 Claude 的資料類型列出來，逐一判斷屬於「可以直接使用」「需要脫敏」還是「不應使用」，作為內部規範的初稿。

    常見問題 Q&A¶

    Q：Anthropic 員工能看到我們的對話內容嗎？ A：Anthropic 員工在技術上有能力存取系統資料，但有嚴格的內部存取控制和審查機制。啟用零保留模式（Zero Retention）後，對話不會保存在 Anthropic 的系統中，員工也無法事後查閱。具體的存取政策詳見 Anthropic 隱私政策。

    Q：我們的對話會不會被用來讓 Claude 回答其他公司的問題？ A：不會。Team 以上方案的對話資料不用於訓練，也就是說你的對話不會影響其他組織使用的模型。每次對話都是獨立的，Claude 不會記住不同客戶之間的資訊。

    Q：GDPR 要求有「被遺忘權」，Anthropic 能配合嗎？ A：可以。Enterprise 方案支援提交資料刪除請求，Anthropic 的 DPO（資料保護長）會在法規要求的時間內處理。啟用零保留模式則能在技術層面上更徹底地滿足這個要求。
