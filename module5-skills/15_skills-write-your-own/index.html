    第十五章：撰寫你的第一個自訂 Skill¶

    設計原則¶

    寫 Skill 不難，但寫得「好用」需要注意幾件事。一個真正有效的 Skill 有三個特點：

    1. Description 精準：Claude 能正確判斷何時觸發
    2. 指令明確：步驟清晰，不模糊
    3. 包含範例：讓 Claude 理解期望的輸出格式

    實作一：Git Commit 格式 Skill¶

    這是最常見、也最實用的入門 Skill，讓 Claude 每次 commit 都自動套用 Conventional Commits 格式：

    mkdir -p ~/.claude/skills/git-commit
cat > ~/.claude/skills/git-commit/SKILL.md << 'SKILLEOF'
---
name: git-commit
description: 當用戶要提交 git commit、commit 程式碼、紀錄變更時觸發
---

# Git Commit 格式規範

## 格式

使用 Conventional Commits 規範：
    ():

    [optional body]

    [optional footer] 

    ## Type 類型

| Type | 說明 | 範例 |
|------|------|------|
| feat | 新功能 | feat(auth): 加入 JWT 登入 |
| fix | 修復 bug | fix(api): 修正 null pointer 錯誤 |
| docs | 文件修改 | docs(readme): 更新安裝步驟 |
| style | 格式調整 | style: 統一縮排為 2 空格 |
| refactor | 重構 | refactor(db): 簡化查詢邏輯 |
| test | 測試 | test(user): 新增登入單元測試 |
| chore | 雜務 | chore: 更新 dependencies |

## 規則

1. description 使用中文
2. 長度不超過 72 字元
3. 使用現在式（「加入」而非「加入了」）
4. 若有多個變更，使用 body 條列說明

## 執行步驟

1. 執行 `git diff --staged` 了解變更內容
2. 根據變更類型選擇 type
3. 撰寫精確的 description
4. 執行 `git commit -m "..."`

    實作二：Email 摘要 Skill¶

    適合需要定期整理郵件的使用者，讓 Claude 每次摘要郵件時都輸出一致的格式：

    mkdir -p ~/.claude/skills/email-summary
cat > ~/.claude/skills/email-summary/SKILL.md << 'SKILLEOF'
---
name: email-summary
description: 當需要分析、摘要或整理電子郵件內容時觸發，包括每日郵件摘要任務
---

# Email 摘要規範

## 輸出格式

每封郵件摘要包含：
1. **寄件者** + 主旨
2. **緊急程度**：🔴 緊急 / 🟡 需回覆 / 🟢 知悉即可
3. **摘要**（50字以內）
4. **建議行動**（選填）

## 分類標準

- 🔴 緊急：需 24 小時內回覆、財務相關、客戶緊急需求
- 🟡 需回覆：一般詢問、會議邀請、需確認事項
- 🟢 知悉即可：通知信、報告、訂閱電子報

## 輸出範例
    📧 Daily Email Summary — 2026/06/22

    🔴 緊急（1） - [王大明] Q3 預算審核 — 需要今天確認數字，請回覆財務部

    🟡 需回覆（3） - [陳小華] 下週產品會議 — 確認 6/26 上午可否出席 - ...

    🟢 知悉即可（5） - [IT部門] 系統維護通知 — 本週六凌晨 2-4 點停機維護 - ...

    Description 撰寫技巧¶

    Description 是 Skill 最關鍵的部分，Claude 靠它判斷要不要觸發這個 Skill。寫得太模糊就不會觸發，寫得太廣又容易誤觸發：

    # ❌ 太模糊
description: 處理程式碼相關任務

# ✅ 具體描述觸發情境
description: 當用戶要進行程式碼審查、review PR、檢查程式碼品質時觸發

# ✅ 包含同義詞
description: 當需要 commit、提交 git、建立 commit message 時觸發

    測試 Skill 是否正確觸發¶

    Skill 寫好後，重啟 Claude Code 對話，用自然語言測試看看：

    $ claude
> 幫我 commit 這些變更
# → 應自動觸發 git-commit skill

> 這個 Skill 沒有自動觸發

# 除錯步驟：
# 1. 檢查 Description 關鍵字是否與你說的話匹配
# 2. 確認 SKILL.md 放在正確路徑
# 3. 重啟 Claude Code 對話

    實作練習¶

    1. 完整建立 git-commit Skill：複製本章的 git-commit Skill 範本，建立到 ~/.claude/skills/git-commit/SKILL.md。在任意有改動的 git 專案中測試：輸入「幫我 commit 這些變更」，確認 Claude 自動觸發 Skill 並使用正確的 Conventional Commits 格式。

    2. 設計你自己的 Skill：根據你實際工作中「每次都要重複告訴 Claude 的事情」，設計一個自訂 Skill。完成後填寫以下檢查清單：

    3. Description 包含足夠的觸發關鍵字
    4. 步驟清晰可執行
    5. 包含至少一個輸出格式範例

    6. 存放路徑正確

    7. Description 優化練習：以下是幾個模糊的 Description，請改寫成精確版本：

    8. 「幫助撰寫文件」→ ___
    9. 「處理錯誤」→ ___
    10. 「分析資料」→ ___

    常見問題 Q&A¶

    Q：Skill 的 Description 應該寫多長？ A：建議一句話到三句話，涵蓋觸發情境和同義關鍵字。太短容易漏觸發，太長則可能與其他 Skill 混淆。例如：「當用戶要提交 git commit、commit 程式碼、建立 commit message、紀錄程式碼變更時觸發」就是理想長度。

    Q：自訂 Skill 建立後需要重啟 Claude Code 嗎？ A：需要。Skills 在對話開始時載入，因此修改或新增 SKILL.md 後，需要結束當前對話並重新啟動（重新執行 claude）才會生效。

    Q：可以有條件觸發嗎？例如「只在特定資料夾才觸發」？ A：可以在 Skill 的描述中加上條件，例如「只對 .py 結尾的檔案觸發」或「當提到 Django 相關任務時觸發」。更精確的條件控制則建議改用專案層級 Skills（只放在特定專案的 .claude/skills/ 中）。
