    第十七章：Skills 在不同產品中的應用搭配¶

    應用矩陣¶

    Skills、MCP、API、Subagent——這些工具不是獨立存在的，真正的威力在於搭配使用。下表是常見場景的工具組合：

    使用場景 工具組合 Skills 的角色
    日常網頁使用 Claude.ai 透過 Projects 設定，作為長期 Custom Instructions
    本地進階使用 Claude Desktop + MCP MCP 提供工具能力；Skills 規範如何使用這些工具
    CLI 開發工作流 Claude Code + Skills 標準化 commit、review、test 等重複性流程
    API 應用開發 Claude API 將 SKILL.md 內容注入 System Prompt
    自動化 Agent Claude Code + MCP + Skills + Subagent 完整 Orchestrator 架構

    搭配一：Claude Desktop + MCP + Skills¶

    MCP 提供「能力」，Skills 提供「規範」，兩者結合讓 Claude 知道能做什麼、也知道該怎麼做：

    Claude Desktop
├── MCP: filesystem     ← 能力：讀寫本地檔案
├── MCP: github         ← 能力：存取 GitHub
└── Skills: email-summary  ← 規範：如何摘要郵件

實際效果：
「幫我整理今天的郵件」
→ 自動觸發 email-summary skill
→ 透過 filesystem MCP 讀取郵件資料
→ 按規範格式輸出摘要

    搭配二：Claude Code + Skills（開發流程標準化）¶

    把開發流程的每個環節都定義好，Claude 就能一路跟著走：

    開發工作流：
1. 開發新功能
   → 自動觸發 feature-dev skill
   → 按 Research → Plan → Implement 三階段進行

2. 完成後提交
   → 觸發 git-commit skill
   → 自動產生符合規範的 commit message

3. 開 PR 前審查
   → 觸發 code-review skill
   → 系統化檢查程式碼品質

4. 部署前安全檢查
   → 觸發 security-audit skill

    搭配三：Skills 作為 API System Prompt¶

    把 Skill 的內容直接注入 API 呼叫的 System Prompt，讓你的應用程式也能享有 Skill 的規範效果：

    import anthropic

# 讀取 Skill 內容
with open('.claude/skills/email-summary/SKILL.md', 'r') as f:
    skill_content = f.read()

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1000,
    system=f"""你是一個郵件摘要助手。

請遵循以下 Skill 規範：

{skill_content}
""",
    messages=[
        {"role": "user", "content": f"請摘要以下郵件：\n\n{email_content}"}
    ]
)

    搭配四：完整 Agent 架構¶

    把所有工具都整合進來，就能建立一個真正的自動化 Agent：

    主 Agent（Claude Code）
│
├── Skills 載入
│   ├── orchestrator-skill  ← 決定如何分派任務
│   ├── email-summary       ← 郵件處理規範
│   └── report-format       ← 報告格式規範
│
├── MCP 工具
│   ├── filesystem          ← 讀寫本地檔案
│   └── outlook-mcp         ← 存取 Outlook
│
└── Subagents 委派
    ├── Subagent A：處理帳號一的郵件
    └── Subagent B：處理帳號二的郵件

    自訂 Skills 的最佳實踐¶

    幾個讓 Skills 真正好用的建議：

    1. 從痛點出發¶

    問自己：「我每次都要重複告訴 Claude 什麼事情？」
這些就是最適合變成 Skill 的內容

    2. 從簡單開始¶

    第一版 Skill：5-10 行指令
持續使用後再迭代改進

    3. 版本控制¶

    # 將 skills 加入 git 管理
git add .claude/skills/
git commit -m "feat: 新增 email-summary skill"

    4. 定期審查¶

    每月檢視哪些 Skill 被使用、哪些沒有觸發
調整 description 或合併冗餘 Skill

    實作練習¶

    1. 完整工作流串接：在 Claude Desktop 中同時啟用 filesystem MCP 和 email-summary Skill（參考本章搭配一的架構），要求 Claude「讀取 Documents/emails.txt 並按 email-summary 格式整理」，體驗 MCP 提供能力、Skill 規範行為的協同效果。

    2. Skills 注入 API System Prompt：複製本章的 Python 範例，將你自訂的 git-commit Skill 內容注入 API 的 system prompt，然後用 API 呼叫請 Claude 分析一段 git diff 並產生 commit message，比較有無注入 Skill 的輸出差異。

    3. 設計你的完整 Agent 架構圖：仿照本章「搭配四：完整 Agent 架構」，畫出你理想中的個人自動化 Agent 架構，包含：主 Agent、需要的 Skills、需要的 MCP 工具、可能的 Subagent 分工。

    常見問題 Q&A¶

    Q：Skills 可以在 Claude API 呼叫中使用嗎？ A：可以，將 SKILL.md 的內容直接嵌入 system prompt 即可（見本章搭配三的範例）。這是在自動化應用中維持一致輸出格式的最有效方式，不需要 Claude Code 環境。

    Q：什麼情況下應該用 Subagent，而非讓主 Agent 循序執行？ A：當子任務之間完全獨立且各自工作量龐大（超過 20 分鐘）時才值得用 Subagent，例如同時分析三個完全不相依的模組。如果任務有先後順序或共用狀態，Subagent 的管理複雜度反而得不償失，循序執行更穩定。

    Q：每月定期審查 Skill 指的是怎麼做？ A：在 Claude Code 對話中詢問「過去這個月你最常用的 Skills 是哪幾個？有哪些從未觸發？」Claude 無法直接追蹤使用頻率，但你可以透過回顧工作記錄（git log 中哪些 commit 格式符合 Skill 規範）來間接評估，並移除長期未使用的 Skill 保持清單整潔。
