    第二十八章：與 Cursor、Copilot 整合實戰¶

    工具不需要二選一¶

    很多開發者在糾結「要用 Claude Code 還是 Cursor」「要用 Claude 還是 Copilot」——但這個問題本身就問錯了。這些工具在開發流程中扮演不同的角色，組合使用才能發揮最大效益。

    這一章的核心觀念是：不同工具負責開發流程的不同階段，而不是用一個取代另一個。

    Claude Code 與 Cursor 的分工¶

    各自的強項¶

    Claude Code（CLI）最擅長： - 跨多個檔案的大規模重構 - 理解整個專案的架構後提出方案 - 執行測試、讀取日誌、根據結果調整 - 完整的工作流自動化（Explore → Plan → Implement → Commit）

    Cursor 最擅長： - 在 GUI 編輯器中即時的 AI 補全 - 針對選取程式碼的快速詢問（Cmd+K） - Composer 多檔案同時修改（適合已知目標的小範圍修改） - 對熟悉 VS Code 介面的使用者體驗更順暢

    搭配使用的工作流¶

    實際開發中，兩者的分工大概長這樣：

    新功能開發流程：

① 用 Claude Code 做前期規劃
  > 我要為這個 API 加入分頁功能，先了解現有結構

  Claude 閱讀相關檔案 → 提出完整實作計畫
  ↓

② 確認計畫後，在 Cursor 中實作細節
  在 Cursor 編輯器中開啟相關檔案
  用 Composer 按照計畫修改多個檔案
  ↓

③ 回到 Claude Code 做整合驗證
  > 請執行測試並確認沒有 regression
  > 幫我寫 commit message

    在 Cursor 中使用 Claude 作為後端模型¶

    Cursor 支援選擇底層 AI 模型，可以切換到 Claude：

    Cursor 設定路徑：
Cursor → Settings → Models → 選擇 claude-sonnet-4-6 或 claude-opus-4-6

    切換到 Claude 後，Cursor 的 Composer 和 Chat 功能就會使用 Claude 的能力，結合了 Cursor 的介面體驗和 Claude 的推理品質。

    注意：使用 Cursor + Claude 需要提供你的 Anthropic API Key（在 Cursor 設定中填入），這部分費用按 API Token 計算，與 Claude.ai 訂閱分開。

    Claude Code 與 GitHub Copilot 的分工¶

    各自的強項¶

    GitHub Copilot 最擅長： - 寫程式時的行內自動補全（真正的「副駕駛」體驗） - 根據你目前游標位置的上下文推測下一行 - 快速生成樣板程式碼（boilerplate） - 深度整合 GitHub PR 流程（Copilot for PRs）

    Claude Code 最擅長： - 需要理解整個專案才能回答的問題 - 複雜的 Bug 診斷（跨多個模組的問題） - 產生完整的功能模組（不只是下一行） - 執行並驗證程式是否正確

    搭配使用的工作流¶

    兩者在開發中的定位：

    日常開發節奏：

邊寫程式 → Copilot 自動補全（即時，不打斷流程）
    ↓
遇到不確定的地方 → Copilot Chat 快速問（小問題）
    ↓
遇到複雜問題 → 切換到 Claude Code（大問題）
    ↓
完成功能 → Claude Code 做整體 Code Review
    ↓
準備 PR → Copilot for PRs 自動生成描述
    ↓
Claude 審閱 PR 內容的技術決策

    避免工具重疊的原則¶

    兩個工具同時訂閱不代表兩個都要一直開著，可以依任務性質切換：

    任務 用哪個
    快速補全目前函數的下一行 Copilot（行內）
    「這個函數應該怎麼優化？」 Copilot Chat 或 Claude
    「這個 Bug 是哪裡出了問題？」（複雜） Claude Code
    「幫我重構這個模組」 Claude Code
    「幫我根據這段程式碼描述寫 PR description」 Copilot for PRs

    MCP 與 Cursor 整合¶

    Cursor 目前正在逐步支援 MCP（Model Context Protocol），讓 Cursor 也能連結外部工具。這和 Claude Code 的 MCP 整合是相同標準，設定方式類似：

    // Cursor 的 .cursor/mcp.json（若支援）
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/你的用戶名/Documents"
      ]
    }
  }
}

    由於 MCP 是開放標準，你在 Claude Code 中設定的 Skills 和工作規範（SKILL.md）可以直接沿用在支援 MCP 的其他工具中，不需要重複設定。

    實際場景示範¶

    場景一：重構大型模組¶

    # Step 1：用 Claude Code 理解現況並規劃
$ claude
> 請分析 src/payment/ 整個模組，
  找出目前架構的問題，並提出重構計畫

[Claude 閱讀所有相關檔案]
[Claude 提出詳細的重構計畫，包含每個步驟的影響範圍]

# Step 2：在 Cursor 的 Composer 中執行已確認的計畫
# 直接讓 Cursor Composer 按照計畫修改指定檔案

# Step 3：回到 Claude Code 驗證
$ claude --continue
> 請執行 payment 相關的測試，確認重構沒有 regression

    場景二：Debug 一個跨模組的問題¶

    # 先用 Copilot Chat 快速問（如果是簡單問題）
# 如果 Copilot 答不上來，切換到 Claude Code

$ claude
> 我的訂單建立 API 在高並發下會出現重複訂單，
  錯誤只有偶發，請幫我分析可能的 race condition 來源

[Claude 讀取相關程式碼、資料庫 schema、事務處理邏輯]
[Claude 系統化找出問題根源並提出修正方案]

    實作練習¶

    1. 工作流設計：在你目前的開發環境中，畫出一個典型功能開發的流程圖，標示出哪些步驟用 Claude Code、哪些用 Cursor / Copilot、哪些兩者都可以。

    2. Cursor + Claude 後端測試：若你有 Cursor，在設定中填入 Anthropic API Key，切換到 Claude 模型，用 Cursor Composer 完成一個小功能，比較和使用 OpenAI 模型時的差異。

    3. 分工實驗：選擇一個中等規模的任務（如：為現有 REST API 新增 WebSocket 支援），完整走一遍「Claude Code 規劃 → Cursor 實作 → Claude Code 驗證」的流程，記錄每個階段花費的時間和體驗感受。

    常見問題 Q&A¶

    Q：Cursor 已經很強了，還需要另外用 Claude Code 嗎？ A：取決於你的工作模式。Cursor 在 GUI 介面和即時補全上體驗很好，但 Claude Code 的優勢是完整的工作流自動化——它能執行指令、讀取日誌、跑測試、根據結果調整，然後 commit，這些 Cursor 目前做不到。如果你只需要編輯器內的 AI 輔助，Cursor 就夠了；如果你想讓 AI 真正「幫你做事」，Claude Code 是必要的。

    Q：同時開著 Copilot 和 Claude Code，電腦會很慢嗎？ A：不會。Copilot 是 VS Code 的輕量擴充套件，只在你打字時才觸發 API 呼叫；Claude Code 是終端機工具，只在你主動輸入指令時才運作。兩者的資源消耗都很低，同時使用對系統效能影響不大。

    Q：SKILL.md 可以在 Cursor 中使用嗎？ A：部分可以。隨著 Cursor 逐步支援 MCP，你在 .claude/skills/ 中設定的 SKILL.md 可以透過 MCP 在 Cursor 中生效。但目前（2026 年）支援程度還在發展中，最穩定的方式仍是在 Claude Code 中使用。官方文件請參考 Cursor 的 MCP 整合說明。
