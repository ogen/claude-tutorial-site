    第二十一章：提示詞工程（Prompt Engineering）¶

    核心技巧¶

    Prompt 寫法沒有唯一答案，但有一些已被驗證有效的模式。這一章介紹五種最常用的技巧，每種都適合不同的場景。

    1. Zero-Shot（零樣本）¶

    最直接的方式，直接描述任務，不給範例。適合簡單任務或你不確定輸出格式時的初次嘗試：

    請分析以下合約條款，指出對甲方不利的條文：
[合約內容]

    2. Few-Shot（少樣本）¶

    提供 2-3 個範例，讓 Claude 理解你要的輸出格式。當 Zero-Shot 的輸出格式不穩定時，加幾個範例通常能解決：

    以下是郵件分類範例：

郵件：「請問明天的會議幾點開始？」
分類：🟡 需回覆

郵件：「系統將於週末進行維護」
分類：🟢 知悉即可

現在請分類：「客戶反映付款頁面無法使用！」
分類：

    3. Chain of Thought（思維鏈）¶

    請 Claude 一步一步說明推理過程，特別適合需要高精度、需要驗證推理的場景：

    請一步一步分析這段程式碼的執行流程，
然後指出可能出現 race condition 的地方。

[程式碼]

    4. XML 標籤結構化¶

    輸入內容有多個不同性質的部分時，用 XML 標籤隔開，Claude 會更容易分辨哪個是任務、哪個是資料：

    <task>
  分析競爭對手的產品策略
</task>

<context>
  我們是一家台灣 B2B SaaS 公司，主要客戶是中小企業
</context>

<competitor_info>
  [競爭對手資料]
</competitor_info>

<output_format>
  請輸出 JSON，包含：
  - strengths（優勢列表）
  - weaknesses（劣勢列表）
  - opportunities（機會列表）
</output_format>

    5. 角色設定（Role Prompting）¶

    給 Claude 一個具體的身份和背景，它的回答會更貼近那個角色的視角和知識範圍：

    你是一位有 15 年經驗的台灣稅務會計師，
熟悉中華民國稅法與電商稅務規定。
接下來我會問你關於個人接案的稅務問題，
請用實務角度回答，並在必要時提醒我諮詢正式會計師。

    輸出格式控制¶

    有時候你需要 Claude 輸出特定格式，以便程式後續處理。

    強制 JSON 輸出¶

    response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system="你只輸出有效的 JSON，不要有任何其他文字或 markdown",
    messages=[{"role": "user", "content": "把以下文字轉成結構化資料：[文字]"}]
)

import json
data = json.loads(response.content[0].text)

    強制特定結構¶

    請嚴格按照以下格式輸出，不要加入其他內容：

標題：[一句話摘要]
緊急度：[高/中/低]
建議行動：[具體步驟]

    常見錯誤與修正¶

    遇到輸出不如預期時，對照這張表先找原因：

    問題 原因 修正方式
    回答太長 未限制長度 加入「不超過 200 字」
    格式不穩定 未給範例 加入 few-shot 範例
    猜測資訊 未說明不確定時怎麼做 加入「不確定請說不知道」
    語言混雜 未指定語言 System prompt 指定語言

    實作練習¶

    1. Zero-Shot 到 Few-Shot 改寫：找一個你常做的分類任務（如：郵件分類、問題類型標記），先用 Zero-Shot 提問，再加入 2-3 個範例改寫成 Few-Shot，比較兩次輸出的格式一致性。

    2. Chain of Thought 應用：選一道有多個步驟的邏輯問題（如：程式 Bug 診斷、合約條款分析），在 Prompt 末尾加上「請一步一步說明你的推理過程」，觀察 CoT 是否讓答案更準確、更易驗證。

    3. 強制 JSON 輸出：設計一個 Prompt 讓 Claude 將以下文字解析成 JSON：「王小明在台北工作，今年32歲，是一名軟體工程師」，要求輸出包含 name、city、age、job 欄位，並用 Python 的 json.loads() 驗證輸出是否為合法 JSON。

    常見問題 Q&A¶

    Q：Chain of Thought 會讓 API 費用增加嗎？ A：會。CoT 要求 Claude 輸出完整的推理過程，output tokens 會顯著增加（有時是無 CoT 的 3-5 倍）。建議只在需要高精度或需要驗證推理過程的場景使用，日常簡單任務用 Zero-Shot 即可。

    Q：XML 標籤一定要用特定名稱嗎？ A：不需要，XML 標籤名稱可以自定義，重要的是語意明確。<document>、<context>、<task>、<output_format> 是常見慣例，但你也可以用 <文件>、<背景>，Claude 都能正確理解。

    Q：如何測試哪種 Prompt 技巧最適合我的任務？ A：建議建立一個「測試集」：準備 10-20 個代表性的輸入，分別測試不同 Prompt 版本的輸出，統計格式正確率、資訊完整率等指標，用數據決定最佳策略，而非直覺判斷。
